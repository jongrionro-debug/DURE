1. 뎁스(Depth)에 따른 분할 (숲과 나무 나누기)

시스템 전체를 그릴 때는 해상도를 다르게 해서 여러 장의 도면을 만듭니다.

레벨 1: 서비스 플로우 (Macro)

목적: 전체적인 사용자의 여정(User Journey)을 보여줍니다.

특징: 디테일한 버튼 클릭이나 예외 처리 로직은 모두 생략합니다. 예를 들어 [앱 실행] → [로그인] → [메인 홈] → [클래스 탐색] → [결제] → [수업 참여]처럼 아주 큰 덩어리만 연결해서 서비스의 전체 청사진을 보여줍니다.

레벨 2: 태스크 플로우 (Micro)

목적: 특정 기능이나 목적을 달성하기 위한 구체적인 흐름입니다.

특징: 현업 기획서(PRD)에 들어가는 가장 일반적인 플로우차트입니다. 전체가 아니라 '회원가입/로그인 플로우', '방과후 수업 결제 플로우', 'AI 리포트 생성 플로우' 등으로 철저하게 쪼개서 해당 기능 내에서의 화면 이동과 조건문만 그립니다.

2. 스윔레인(Swimlane) 다이어그램 활용 (역할별 분리)

플랫폼에는 아이, 학부모, 강사, 운영자처럼 역할(Role)이 다양한 경우가 많습니다. 이들이 어떻게 상호작용하는지 한눈에 보려면 수영장 레인처럼 구역을 나눈 스윔레인 다이어그램을 씁니다.

User 레인: 학부모가 출석 알림을 확인하고 버튼을 누른다.

Front-end 레인: 알림 내역 UI를 렌더링하고 서버에 상태 값을 요청한다.

Back-end/AI 레인: AI 모델이 활동 데이터를 요약하여 리포트 데이터를 생성한다.
이렇게 레인을 나누면 시스템이 아무리 커져도 '누가, 어느 시점에, 무슨 행동을 하는지' 추적하기가 매우 직관적입니다.

3. 컴포넌트화 및 모듈화 (반복되는 플로우 빼기)

개발에서 공통 함수를 모듈로 빼서 재사용하듯, 기획에서도 자주 쓰이는 흐름은 **'공통 모듈'**로 취급합니다.

예를 들어 '로그인', '휴대폰 본인인증', '결제 모듈'은 시스템 곳곳에서 쓰입니다.

이걸 매번 플로우차트에 그리지 않고, [본인인증 프로세스 A 진행] 이라는 하나의 블랙박스(단일 프로세스 박스)로 둔 뒤, "상세 플로우는 A 문서를 참고하세요"라고 링크를 걸어버립니다.

4. 시각화의 한계 인정 (글과 정책으로 보완)

모든 예외 상황(Unhappy Path)을 다 화살표로 빼서 그리면 도면이 스파게티가 됩니다.

**해피 패스(정상적인 성공 흐름)**를 중심으로 메인 뼈대를 아주 깔끔하게 그립니다.

비밀번호 5회 오류, 네트워크 단절, 잔액 부족 등 자잘한 예외 케이스는 마름모꼴 화살표로 다 빼는 대신, PRD 하단에 **[예외 처리 정책 테이블]**을 만들어 글로 명시합니다. 개발자들도 얽힌 선을 따라가는 것보다 표를 보고 예외 처리를 코딩하는 것을 훨씬 편해합니다.
